# Database Schema

## Entity: Task

The application uses a single `Task` table in PostgreSQL, managed by Prisma ORM.

### Table Structure

```sql
CREATE TABLE "Task" (
    "id"          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    "title"       VARCHAR(200) NOT NULL,
    "description" VARCHAR(2000),
    "status"      "TaskStatus"   NOT NULL DEFAULT 'TODO',
    "priority"    "TaskPriority" NOT NULL DEFAULT 'MEDIUM',
    "dueDate"     TIMESTAMPTZ,
    "createdAt"   TIMESTAMPTZ  NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updatedAt"   TIMESTAMPTZ  NOT NULL
);
```

### Field Descriptions

| Column       | Type           | Nullable | Default   | Description                            |
|-------------|----------------|----------|-----------|----------------------------------------|
| `id`        | UUID           | No       | Auto-gen  | Primary key, generated by PostgreSQL   |
| `title`     | VARCHAR(200)   | No       | вҖ”         | Task title, 1вҖ“200 characters           |
| `description`| VARCHAR(2000) | Yes      | NULL      | Optional detailed description          |
| `status`    | Enum           | No       | `TODO`    | Workflow state: TODO, IN_PROGRESS, DONE|
| `priority`  | Enum           | No       | `MEDIUM`  | Urgency level: LOW, MEDIUM, HIGH       |
| `dueDate`   | TIMESTAMPTZ    | Yes      | NULL      | Optional deadline                      |
| `createdAt` | TIMESTAMPTZ    | No       | now()     | Auto-set on row creation               |
| `updatedAt` | TIMESTAMPTZ    | No       | Auto      | Auto-updated by Prisma on modification |

### Enums

```sql
CREATE TYPE "TaskStatus" AS ENUM ('TODO', 'IN_PROGRESS', 'DONE');
CREATE TYPE "TaskPriority" AS ENUM ('LOW', 'MEDIUM', 'HIGH');
```

**Why DB-level enums?** PostgreSQL enums enforce data integrity at the storage level. Invalid values are rejected before they ever reach the application, providing a safety net beyond Zod validation.

### Indexes

```sql
CREATE INDEX "Task_status_idx" ON "Task"("status");
CREATE INDEX "Task_dueDate_idx" ON "Task"("dueDate");
```

| Index              | Column   | Rationale                                            |
|--------------------|----------|------------------------------------------------------|
| `Task_status_idx`  | `status` | The primary list view filters by status. Without this, every filtered query full-table-scans. |
| `Task_dueDate_idx` | `dueDate`| Sorting by due date is a supported query parameter. Also enables efficient "overdue" range queries. |

**Why not index `createdAt`?** PostgreSQL stores rows roughly in insertion order, and the primary key (UUID) provides sufficient uniqueness. For a table with <100k rows, the default sort-by-createdAt query performs well without a dedicated index.

### Prisma Schema Location

The schema is defined in `backend/prisma/schema.prisma` and serves as the single source of truth for the database structure. Migrations are generated from this file.
